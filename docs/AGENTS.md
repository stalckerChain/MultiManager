# AGENTS.md — Правила работы для ИИ-агентов

> **Проект:** MultiManager — AI-Driven Web Automation Platform
> **Стек:** Node.js (Express, better-sqlite3) + Electron (Vue 3, Ant Design Vue)
> **Тесты:** Vitest, `npm test`

---

## 1. Жизненный цикл задачи

Каждая задача проходит строго определённый цикл. Нарушать порядок нельзя.

### Шаг 1. План

Получив задачу, агент **не пишет код**, а составляет план:

1. Анализирует текущее состояние (файлы, тесты, зависимость от других модулей).
2. Определяет конкретные изменения: какие файлы, что менять, какая логика.
3. Оценивает риски: затрагивает ли задача API, БД, безопасность, GUI.
4. Предлагает план пользователю на согласование.

Формат плана — свободный, но обязан содержать:
- Что делаем (конкретные пункты)
- Какие файлы затрагиваются
- Какие риски / критичные части системы затрагиваются (если есть)

### Шаг 2. Согласование плана

- Пользователь одобряет план или запрашивает доработку.
- Если план не одобрен — дорабатываем до получения одобрения.
- **Если изменения затрагивают критичные части системы** (безопасность, миграции БД, API-контракты) — это оговаривается отдельно после составления плана.

### Шаг 3. Фиксация в TASK.md

После одобрения плана в файл `TASK.md` записывается:
- Текущая задача (что делаем)
- Подробный план по реализации (пошагово)

`TASK.md` — это контрольный документ. На нём ссылается весь процесс.

### Шаг 4. Реализация

- Пишем код согласно плану.
- Запускаем `npm test` после каждого изменения.
- Если тесты падают — исправляем немедленно.

### Шаг 5. Сборка и проверка

- Собираем инсталлер и портабл-версию.
- Пользователь проверяет работоспособность.

### Шаг 6. Финализация (если задача принята)

После одобрения сборки выполняем последовательно:

1. Обновляем документацию в `docs/` (API.md, DATABASE.md и т.д. при необходимости).
2. Пишем тесты, покрывающие реализованную задачу.
3. Обновляем `README.md` (раздел «Что нового»).
4. Обновляем `CHANGELOG.md`.
5. Если нужно — обновляем `TS.md` (техническое задание).
6. Создаём коммит.
7. Сообщаем пользователю результат.

### Шаг 7. Возврат (если задача не принята)

- Если пользователь не принимает результат — возвращаемся к шагу 1.
- Составляем новый план с учётом замечаний.
- Показываем план → согласование → и далее по циклу.

## 2. Что нужно уточнять у пользователя

### Всегда спрашивать:

| Ситуация | Что уточнять |
|----------|-------------|
| Задача сформулирована размыто | Конкретику: какой файл, какое поведение, ожидаемый результат |
| Несколько вариантов решения | Предложить варианты с trade-offs, дать выбор |
| Изменение API-контракта | Версионирование, backward compatibility |
| Изменение схемы БД | Миграция, совместимость со старыми данными |
| Добавление новой зависимости | Обоснование, альтернативы, размер бандла |
| Удаление кода | Убедиться что не используется (grep callers) |
| Security-вопросы | Всегда эскалировать на пользователя |
| Масштабирование | Оценка производительности на 100+ профилей |

### НЕ спрашивать (делать сразу):

- Исправление очевидных багов (null check, off-by-one, typo)
- Добавление тестов для уже написанного кода
- Обновление зависимостей с security-patch
- Форматирование и стилизация по существующему конвеншену

## 3. Конвенции кода

### JavaScript (Backend: `src/`)

- **Стиль:** CommonJS (`require`/`module.exports`), без ESM в backend.
- **Именование:** camelCase для переменных/функций, PascalCase для классов.
- **Логирование:** `pino` через `require('../core/logger')`. Не `console.log`.
- **Ошибки:** `throw new Error('...')` с информативным сообщением. Не проглатывать ошибки.
- **Асинхронность:** `async/await`, не callback-стиль.
- **БД-запросы:** через query-объекты в `src/db/queries.js`. Не писать SQL в роутерах.

### Vue 3 (Frontend: `gui/src/renderer/`)

- **API:** Composition API (`<script setup>`), не Options API.
- **UI-компоненты:** Ant Design Vue (`a-button`, `a-table`, и т.д.).
- **Стили:** Tailwind CSS. Не писать inline-стили.
- **State:** Pinia stores.
- **i18n:** Все строки через `t('key')`. Не хардкодить текст.

### Тесты (`tests/`)

- **Фреймворк:** Vitest (`describe`/`it`/`expect`).
- **Именование:** `*.test.js` рядом с тестируемым файлом.
- **Паттерн:** Arrange → Act → Assert.
- **Моки:** `vi.mock()`, `vi.fn()`. Не мокать то, что можно реально вызвать.
- **Интеграционные тесты:** `tests/integration/`, требуют запущенный бэкенд.

## 5. Безопасность

1. **Никогда** не логировать пароли, токены, прокси-credentials, master key.
2. **Все mutating-эндпоинты** (POST/PUT/DELETE) к `/api/profiles`, `/api/proxies`, `/api/cookies` требуют master key gate (`hasMasterKey()`).
3. **WebSocket** требует `?token=` query parameter.
4. **CDP-инжект:** использовать `Runtime.callFunctionOn`, не конкатенировать JS-строки.
5. **Прокси:** валидировать scheme, блокировать private/local адреса (SSRF protection).
6. **Файлы:** всегда удалять temp-файлы в `finally` block.
7. **Рекавери-ключ** показывается один раз и удаляется из БД.

## 6. Документация

- Все документы в `docs/` ведутся на **русском языке** (основной), с переводами `*.en.md` и `*.zh.md` при необходимости.
- **API.md** — референс всех эндпоинтов. Обновлять при добавлении/изменении API.
- **DATABASE.md** — схема + миграции. Обновлять при изменении схемы.
- **CHANGELOG.md** —angelog по версиям. Обновлять при каждом релизе.
- **TASK.md** — текущая задача + план реализации.
- **TS.md** — техническое задание. Обновлять при изменении архитектуры или требований.

## 7. Git и коммиты

- **Формат:** `<type>(<scope>): <description>`
  - `type`: feat, fix, security, refactor, test, docs, chore
  - `scope`: api, db, gui, crypto, proxy, executor, multi-control
- **Примеры:**
  - `feat(api): add cookie bulk import endpoint`
  - `fix(db): handle migration for missing columns`
  - `security(crypto): rotate core token on each start`
- **Не коммитить:** `.env`, credentials, temp-файлы, `node_modules/`.

## 8. Эскалация

Агент **обязан** остановиться и обратиться к пользователю, если:

1. Задача противоречит текущей архитектуре (см. TS.md).
2. Требуется изменение security-модели (авторизация, шифрование, master key).
3. Требуется миграция БД с потенциальной потерей данных.
4. Обнаружен конфликт с другим компонентом (GUI ↔ Backend ↔ Python).
5. Нет тестов и невозможно быстро проверить корректность.
6. Задача выходит за рамки одного модуля и затрагивает 3+ файла.

## 9. Полезные команды

```bash
# Тесты
npm test                        # Все unit-тесты
npm run test:api                # Интеграционные тесты (требует запущенный бэкенд)
npm run test:all                # Всё вместе

# Разработка
npm run dev                     # Watch-режим бэкенда
cd gui && npm run dev           # Watch-режим GUI

# Сборка
cd gui && npm run build         # Сборка GUI

# Линтинг
npm run lint                    # ESLint для src/

# Версионирование
npm run bump                    # Автобамп версии
```
