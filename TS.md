## SOFTWARE REQUIREMENTS SPECIFICATION (SRS) / ТЕХНИЧЕСКОЕ ЗАДАНИЕ## Кросссплатформенный антидетек-браузер с API для ИИ (MVP аналог AdsPower)
------------------------------
## 1. Общие сведения и архитектура системы
Продукт предназначен для создания, хранения и запуска изолированных профилей браузера на базе кастомного C++ ядра CloakBrowser.
Архитектура системы жестко разделена на два независимых слоя:

   1. Core-движок (Бэкенд): Консольное Node.js-приложение, работающее в фоновом режиме операционной системы. Предоставляет локальный REST API для управления базой данных и жизненным циклом процессов браузера.
   2. GUI (Фронтенд): Графическая десктопная оболочка (Electron или Tauri), выполняющая роль визуального интерфейса. GUI коммуницирует с Core-движком исключительно через локальные HTTP-запросы.

Система является кроссплатформенной и должна стабильно работать на Windows 11, macOS и Linux.
------------------------------
## 2. Безопасность и авторизация локального API

* Локальный хост: Core-движок открывает защищенный порт только на локальном интерфейсе (127.0.0.1:3000). Внешние сетевые запросы из интернета блокируются брандмауэром.
* Механизм обмена ключами (Handshake): Во избежание утечки токена с диска, GUI-процесс (Electron) при запуске Core-движка через child_process.fork генерирует случайный криптографический ключ в оперативной памяти и передает его бэкенду в качестве аргумента запуска (флаг --api-token=SECRET_VALUE).
* Авторизация запросов: Все входящие HTTP-запросы к Core-движку (как от GUI, так и от внешних ИИ-агентов) должны содержать заголовок Authorization: Bearer SECRET_VALUE. При отсутствии или несовпадении токена API возвращает статус 401 Unauthorized.
* Доступ для ИИ-агентов: Секретный токен должен быть доступен для копирования в окне настроек GUI, чтобы пользователь мог передать его в конфигурацию своего ИИ-скрипта.

------------------------------
## 3. Хранение данных (База данных и локальные файлы)

* Тип БД: Все системные настройки, метаданные и параметры отпечатков хранятся в SQLite (через производительную нативную библиотеку better-sqlite3).
* Обеспечение надежности: Использование режима WAL (Write-Ahead Logging) в SQLite и ACID-транзакций для полной защиты от повреждения данных при аварийном выключении ПК или контейнера.
* Директория приложения: Все файлы (БД app.db, кэш, логи) сохраняются в защищенную домашнюю папку пользователя:
* Windows: AppData/Roaming/CloakManager/
   * macOS: Library/Application Support/CloakManager/
   * Linux: .config/CloakManager/

## Структура схемы БД профиля:

* id (UUIDv4) — уникальный ключ профиля для API.
* number (INTEGER) — инкрементный порядковый номер для визуального удобства.
* name (TEXT) — пользовательское имя аккаунта.
* proxy_id (INTEGER) — внешний ключ, связывающий профиль с таблицей менеджера прокси.
* fingerprint_seed (TEXT) — уникальное зерно для генерации стабильного шума Canvas/WebGL внутри CloakBrowser.
* platform (TEXT: windows | macos | linux).
* user_agent, screen_resolution (TEXT).
* hardware_cores, hardware_memory (INTEGER).
* extensions (TEXT, JSON-массив идентификаторов используемых расширений).
* status (TEXT: stopped | starting | running).

## 4. Функциональные модули системы## 4.1. Генератор случайных отпечатков (Fingerprint Generator)

* Модуль генерации реалистичных цифровых личностей «в один клик».
* На основе выбранной platform система сопоставляет параметры, исключая логические аномалии (например, недопустимо генерировать Safari User-Agent для платформы Windows или выставлять 32 ядра процессора для конфигурации бюджетного ноутбука).
* Автоматически генерирует уникальный fingerprint_seed для маскировки Canvas/WebGL на уровне C++ ядра CloakBrowser.

## 4.2. Менеджер прокси и ротация (Proxy Manager)

* Централизованная таблица прокси (HTTP, HTTPS, SOCKS5) с возможностью пакетного импорта.
* Поддержка ротации (Мобильные прокси): В свойствах прокси закладывается поле proxy_rotation_url. Если оно заполнено, перед стартом профиля Core-движок выполняет асинхронный HTTP GET-запрос к API провайдера для смены IP, ожидает status 200 и выдерживает технологическую паузу в 3 секунды.
* Proxy Checker: После ротации (или для обычного прокси) движок делает тестовый запрос к https://ipify.org. Если прокси не отвечает или выдает ошибку, запуск браузера блокируется, статус профиля переводится в stopped, а API возвращает ошибку 412 Precondition Failed с выводом предупреждения в GUI.

## 4.3. Управление куки (Cookie Import/Export)

* API-эндпоинты для импорта/экспорта файлов куки в форматах JSON и Netscape (TXT).
* Перед запуском процесса Core-движок раскладывает куки в изолированную директорию профиля --user-data-dir.

## 4.4. Логика синхронизатора (Multi-Control)

* Модуль трансляции мыши и клавиатуры из «главного» окна (master) в «ведомые» окна (slave) через Chrome DevTools Protocol (CDP).
* Throttling: Координаты мыши транслируются с ограничением частоты (каждые 25 мс) для минимизации нагрузки на CPU хост-системы.
* **Синхронизация вкладок (Tab Sync):**
  * **Tab Mapping 1:N** — каждый таб мастера маппится в соответствующие табы слейвов через структуру `Map<masterTargetId, Map<slaveId, slaveTargetId>>`.
  * **Ctrl+T (Новая вкладка):** Native hook перехватывает Ctrl+T на уровне ОС. Браузер мастера открывает вкладку нативно (без CDP). HTTP polling `/json` (DevTools endpoint) обнаруживает новый таб в течение ≤300 мс и синхронизирует его во все слейвы: attach мастера → createTab + attach в слейвах → mapTab → setActiveMasterTab.
  * **Ctrl+W (Закрыть вкладку):** Native hook перехватывает Ctrl+W. Бэкенд закрывает соответствующие табы слейвов через CDP `Target.closeTarget` и удаляет маппинг. Браузер мастера закрывает вкладку нативно.
  * **_blank / window.open:** Ссылки с `target="_blank"` не перехватываются в sync-script. HTTP `/json` polling (300 мс) обнаруживает новый таб в мастере, после чего `syncNewMasterTab` создаёт аналогичные табы во всех слейвах, attach'ит их и маппит.
  * **Навигация:** При переходе мастера на новый URL (Page.frameNavigated), бэкенд навигирует соответствующий замапленный таб в каждом слейве через `Page.navigate`. Если маппинг отсутствует — fallback на первый/дефолтный таб слейва.
  * **Активация вкладок:** CDP событие `Target.targetInfoChanged` вызывает `setActiveMasterTab` и `_syncActiveTabToSlaves`, активируя соответствующие табы слейвов.
  * **Источник вкладок — HTTP /json polling** (не WS `Target.setAutoAttach`): Антидетект-браузер не экспортирует нативно-открытые табы через CDP WebSocket. Единственный надёжный источник — HTTP DevTools endpoint `GET /json` (опрашивается каждые 300 мс).
  * **Фокус окон при старте:** После активации синхронизации все окна слейвов последовательно выводятся на передний план (через OS-level `SetForegroundWindow` / `xdotool windowactivate`), затем окно мастера. Это гарантирует, что пользователь видит весь набор синхронизированных окон после нажатия кнопки «Sync». Эндпоинт: `POST /api/multi-control/focus-windows`.

## 4.5. Автоматизация ввода для ИИ (Human-like Typing)

* Специализированный метод API для ввода текста ИИ-агентом.
* Вместо мгновенной вставки значения, Core-движок через CDP эмулирует последовательное нажатие клавиш со случайной задержкой от 50 до 150 мс на символ, включая редкие случайные опечатки и их последующее удаление через Backspace.

## 4.6. Безопасная очистка дискового пространства

* Метод очистки временных файлов профиля (Cache, Code Cache, GPUCache).
* Блокировка состояния (Mutex): Очистка файлов физически заблокирована, если статус профиля в БД равен starting или running. Запрос на очистку обрабатывается только при статусе stopped. При попытке очистить кэш активного браузера API возвращает 409 Conflict.

## 4.7. Кроссплатформенный контроль процессов (Anti-Zombie)

* При старте CloakBrowser Core-движок перехватывает и сохраняет его системный PID.
* Команда STOP инициализирует штатное закрытие. Если в течение 5 секунд процесс не завершился, Core принудительно уничтожает его на уровне ОС:
* На macOS/Linux: Системный вызов process.kill(pid, 'SIGKILL').
   * На Windows: Системная команда через taskkill /F /PID <pid> (или использование Node-библиотеки tree-kill).

------------------------------
## 5. Стратегия логирования (Logging)
Система ведет раздельные логи с использованием легковесного логгера (pino):

   1. Системный лог (core.log): Фиксирует запуск API-сервера, ошибки подключения к SQLite, генерацию токенов и общие сбои бэкенда.
   2. Лог профиля (logs/profile_[ID].log): Каждый профиль имеет свой изолированный файл лога, куда записываются этапы ротации прокси, результаты проверки Proxy Checker, ошибки запуска бинарника CloakBrowser и логи сессий автоматизации ИИ.

------------------------------
## 6. Стратегия тестирования (Testing)
Для обеспечения стабильности MVP внедряется автоматическое тестирование (фреймворк Vitest):

* Unit-тесты: Валидация парсеров куки (JSON/Netscape), проверка генератора отпечатков на отсутствие логических аномалий, проверка middleware авторизации по токену.
* Integration-тесты: Тестирование изоляции SQLite (параллельная запись из разных потоков), эмуляция падения прокси с проверкой корректного срабатывания блокировки запуска, а также тест-скрипт полного цикла: запуск пустого CloakBrowser -> перехват его PID -> проверка доступности возвращаемого ws_endpoint -> принудительный kill процесса.

------------------------------
## 7. Формат ответа API для интеграции с ИИ
При успешной валидации прокси и запуске CloakBrowser, Core-движок возвращает ИИ-агенту JSON-ответ следующего формата:

{
  "status": "success",
  "profile_id": "8f3b201a-cb41-4c12-8671-50e50f3b4d11",
  "pid": 14208,
  "ws_endpoint": "ws://127.0.0.1:3000/devtools/browser/6b9a84ed-733d-4cbf-a9f4-8845fcda942b"
}

Интерпретация: ИИ-агент считывает ws_endpoint и подключает к нему свою библиотеку (Puppeteer/Playwright) для выполнения автономных задач в браузере.

## РАЗДЕЛ: ГРАФИЧЕСКИЙ ИНТЕРФЕЙС ПОЛЬЗОВАТЕЛЯ (GUI)## 1. Технологический стек и архитектура интерфейса

* Платформа: Electron.js [3.2].
* Фронтенд-фреймворк: Vue.js или React [3.2].
* Библиотека компонентов: Ant Design, Shadcn/ui или Tailwind CSS (для быстрого построения таблиц, модальных окон и форм) [3.2].
* Интеграция с Core: GUI общается с фоновым Core-процессом через локальные HTTP-запросы (порт 127.0.0.1:3000) и WebSocket-соединение для мгновенного обновления статусов [3.2].

## 2. Функциональные экраны и модули GUI## 2.1. Экран «Менеджер профилей» (Profile Manager)
Главный рабочий экран приложения, разделенный на три зоны:

   1. Верхняя панель действий (Toolbar):
   * Кнопка «Создать профиль» + выпадающее меню «В 1 клик» (мгновенный запуск генератора случайных отпечатков) [3.3].
      * Кнопка активации «Синхронизатора» [3.3].
      * Панель групповых операций (активируется при выборе строк галочками): Массовый старт, Массовый стоп, Удалить, Очистить кэш [3.3].
      * Поле быстрого поиска по имени/ID и выпадающий список фильтрации по цветным тегам.
   2. Центральная таблица профилей (Реактивная):
   * Столбцы: Чекбокс, Порядковый №, Название + Теги, Прокси (с флагом страны, текущим IP и кнопкой ручной ротации), Конфигурация отпечатка (ОС, версия Chrome, ОЗУ, ядра), Статус (Спит / Запуск... / Работает + PID процесса), Кнопка СТАРТ/СТОП, Кнопка «...» (контекстные действия) [3.3].
      * WebSocket-логика: Если профиль запускается удаленно через API ИИ-агентом, строка профиля должна автоматически изменить статус на Работает и загореться зеленым цветом без перезагрузки интерфейса.
   3. Окно создания и редактирования профиля (Модальное):
   * Вкладка «Основные»: Имя, теги, переключатель ОС (Windows/Mac/Linux), кнопка «Сгенерировать отпечаток» (автозаполнение полей User-Agent, Разрешение, Ядра, ОЗУ случайными валидными данными) [3.3].
      * Вкладка «Прокси»: Выпадающий список выбора из сохраненных прокси или поля прямого ввода нового прокси [3.3].
      * Вкладка «Дополнительно»: Список чекбоксов для выбора расширений, текстовое поле для заметок.
   
## 2.2. Модуль «Window Arranger» (Управление окнами и Сеткой)
Интегрирован в интерфейс Синхронизатора. Управляет физическим положением окон CloakBrowser на мониторе пользователя:

* Режим «Сетка» (Grid): Кнопка, при нажатии которой Electron запрашивает рабочую область монитора хост-системы, делит её на равные части и автоматически расставляет все запущенные окна профилей стык в стык без перекрытий.
* Режим «Перекрытие» (Cascade): Выводит выбранное Главное (Master) окно на передний план по центру экрана. Все остальные ведомые окна выстраивает ровной стопкой («внахлест») строго за ним со смещением (например, на 30px вправо и вниз), формируя визуальную видимость краев всей «колоды» окон и возможность быстрой смены Главного окна.

## 2.3. Экран «Менеджер прокси» (Proxy Manager)

* Таблица прокси-серверов: Отображает статус (Живой/Мертвый/Проверка...), ГЕО (иконка флага страны + город), тип протокола (HTTP/SOCKS5), данные подключения, ссылку ротации, пинг в миллисекундах и счетчик привязанных к нему профилей.
* Глобальные кнопки: Добавить прокси, Проверить выбранные, Удалить неиспользуемые.
* Форма пакетного добавления (Smart Parsing): Большое текстовое поле (Textarea), принимающее список прокси «пачкой». Интерфейс должен автоматически распознавать популярные форматы строк (например, IP:Port:User:Pass или User:Pass@IP:Port), разбивать их на составляющие и распределять по полям базы данных.

## 2.4. Модуль импорта/экспорта куки (Cookie Manager)

* Форма Drag-and-Drop: Всплывающее окно в контекстном меню профиля для перетаскивания файлов куки (.json или .txt Netscape).
* Пре-валидатор фронтенда: Сразу после перетаскивания файла (до сохранения) интерфейс выводит сводку: «Обнаружено куки: X шт. для сайтов: google.com, facebook.com. Статус: Валидны».
* Экспорт: Кнопка «Скачать куки» в строке профиля для мгновенного скачивания текущей сессии в JSON-файл в папку «Загрузки» операционной системы.

## 2.5. Экран расширений (Extensions Manager)

* Интерфейс в виде сетки карточек установленных расширений Chrome. Каждая карточка содержит имя расширения, иконку и глобальный переключатель (Toggle) активности. Добавление новых расширений происходит через загрузку .zip архивов или выбор распакованной папки на диске Windows.

## 2.6. Мониторинг логов и статуса API

* Панель разработчика: Вкладка в интерфейсе, отображающая бегущую строку логов бэкенда (core.log) в реальном времени для визуальной отладки.
* Статус-бар (Футер приложения): Выводит текущий статус локального сервера, порт и кнопку с иконкой «глаза» для просмотра/копирования секретного AUTH_TOKEN, необходимого для авторизации внешних ИИ-агентов [3.2].


## 2.7. Модуль локализации и мультиязычности (i18n / Localization)

* Технологический стек: Использование библиотеки i18next (или аналога под выбранный фреймворк) для бесшовного переключения языков «на лету» без перезагрузки интерфейса.
* Поддерживаемые языки на старте (MVP):
1. English (По умолчанию) — файл en.json
   2. Русский — файл ru.json
   3. 简体中文 (Китайский упрощенный) — файл zh.json
* Архитектура хранения: Все текстовые строки интерфейса (названия кнопок, заголовки таблиц, сообщения об ошибках, плейсхолдеры полей ввода) строго запрещено писать текстом в коде. Они должны вызываться через переменные-ключи (например, t('buttons.start')).
* Интерфейсные элементы:
* В верхний правый угол GUI (или в настройки приложения) добавляется выпадающее меню (Dropdown) выбора языка с отображением иконок флагов или названий языков на их родном наречии (English, Русский, 中文).
* Сохранение выбора: Выбранный пользователем язык должен автоматически сохраняться в локальные настройки SQLite базы данных, чтобы при следующем запуске приложения интерфейс сразу открывался на нужном языке.
* Локализация ошибок бэкенда: Ошибки, возвращаемые Core-движком по API (например, Proxy Connection Refused), должны передаваться в виде кодов ошибок (например, ERR_PROXY_REFUSED), чтобы фронтенд мог сам перевести эту ошибку на выбранный пользователем язык.

## ДОПОЛНЕНИЕ К РАЗДЕЛУ «ГРАФИЧЕСКИЙ ИНТЕРФЕЙС» (GUI)## 3. Системная интеграция, отказоустойчивость и UX## 3.1. Модуль кастомизации тем оформления (Theme Switcher)

* Реализация: Полная поддержка динамической смены тем оформления (Тёмная / Светлая / Системная) с использованием CSS-переменных (дизайн-токенов Tailwind/AntD). [1] 
* Синхронизация: GUI должен уметь отслеживать системные настройки ОС через медиа-запрос prefers-color-scheme и встроенное API Electron nativeTheme. [2] 
* Сохранение состояния: Выбранный режим (Dark/Light/System) записывается в SQLite, чтобы исключить мерцание интерфейса при холодном старте приложения.

## 3.2. Логика автозапуска Core и обработка конфликтов портов

* Проблема: По умолчанию Core пытается занять порт 3000. Если порт занят другим софтом, бэкенд упадет с ошибкой EADDRINUSE.
* Алгоритм обработки:
1. При инициализации child_process.fork главный процесс Electron проверяет доступность порта 3000 (через тестовое net-подключение).
   2. Если порт занят, Electron начинает инкрементный поиск свободного порта в диапазоне 3000–3100 (3001, 3002...).
   3. Найденный свободный порт (например, 3005) Electron принудительно передает бэкенду в качестве аргумента запуска (--port=3005).
   4. Фронтенд автоматически перенаправляет все свои внутренние HTTP и WebSocket запросы на этот динамически выделенный порт.

## 3.3. Устойчивость WebSocket-соединения (Auto-Reconnect)

* Реализация: Логика WebSocket-клиента в GUI должна иметь встроенный механизм Exponential Backoff (экспоненциальное увеличение задержки) для повторного подключения.
* Алгоритм: При непредвиденном разрыве связи с Core (например, бэкенд аварийно перезапустился) GUI не должен выводить критическую ошибку. Он переходит в режим ожидания и пытается переподключиться через 1, 2, 4, 8 секунд. При восстановлении связи интерфейс незаметно для пользователя запрашивает актуальный статус запущенных профилей из SQLite и обновляет таблицу.

## 4.4. Жизненный цикл окна и минимизация в системный трей (System Tray)

* Реализация: Для обеспечения непрерывной работы ИИ-агентов закрытие главного окна приложения не должно завершать процесс.
* Логика работы:
* При запуске создается иконка приложения в системном трее (Tray) Windows/macOS с контекстным меню («Открыть панель», «Статус API», «Выход»).
   * Событие клика по кнопке закрытия окна (close) перехватывается Electron. Окно физически не уничтожается, а скрывается (window.hide()). Процессы GUI и фонового Core продолжают работу.
   * Полное завершение приложения (App Quit) с остановкой всех ИИ-сессий возможно только через явный выбор пункта «Выход» в меню системного трея.

## 3.5. Механизм автоматических обновлений (Auto-Update)

* Реализация: Интеграция стандартного модуля electron-updater в связке со сборщиком electron-builder.
* Логика: При старте приложения GUI отправляет скрытый запрос к вашему удаленному репозиторию (или GitHub Releases) для проверки файла latest.yml. Если доступна новая версия приложения, GUI скачивает её в фоновом режиме. После скачивания пользователю выводится ненавязчивое уведомление: «Доступно обновление. Установить сейчас (перезагрузить приложение) или после закрытия?».
