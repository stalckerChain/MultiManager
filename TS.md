## SOFTWARE REQUIREMENTS SPECIFICATION (SRS) / ТЕХНИЧЕСКОЕ ЗАДАНИЕ## Кросссплатформенный антидетек-браузер с API для ИИ (MVP аналог AdsPower)
------------------------------
## 1. Общие сведения и архитектура системы
Продукт предназначен для создания, хранения и запуска изолированных профилей браузера на базе кастомного C++ ядра CloakBrowser.
Архитектура системы жестко разделена на два независимых слоя:

   1. Core-движок (Бэкенд): Консольное Node.js-приложение, работающее в фоновом режиме операционной системы. Предоставляет локальный REST API для управления базой данных и жизненным циклом процессов браузера.
   2. GUI (Фронтенд): Графическая десктопная оболочка (Electron или Tauri), выполняющая роль визуального интерфейса. GUI коммуницирует с Core-движком исключительно через локальные HTTP-запросы.

Система является кроссплатформенной и должна стабильно работать на Windows 11, macOS и Linux.
------------------------------
## 2. Безопасность и авторизация локального API

* Локальный хост: Core-движок открывает защищенный порт только на локальном интерфейсе (127.0.0.1:3000). Внешние сетевые запросы из интернета блокируются брандмауэром.
* Механизм обмена ключами (Handshake): Во избежание утечки токена с диска, GUI-процесс (Electron) при запуске Core-движка через child_process.fork генерирует случайный криптографический ключ в оперативной памяти и передает его бэкенду в качестве аргумента запуска (флаг --api-token=SECRET_VALUE).
* Авторизация запросов: Все входящие HTTP-запросы к Core-движку (как от GUI, так и от внешних ИИ-агентов) должны содержать заголовок Authorization: Bearer SECRET_VALUE. При отсутствии или несовпадении токена API возвращает статус 401 Unauthorized.
* Доступ для ИИ-агентов: Секретный токен должен быть доступен для копирования в окне настроек GUI, чтобы пользователь мог передать его в конфигурацию своего ИИ-скрипта.

------------------------------
## 3. Хранение данных (База данных и локальные файлы)

* Тип БД: Все системные настройки, метаданные и параметры отпечатков хранятся в SQLite (через производительную нативную библиотеку better-sqlite3).
* Обеспечение надежности: Использование режима WAL (Write-Ahead Logging) в SQLite и ACID-транзакций для полной защиты от повреждения данных при аварийном выключении ПК или контейнера.
* Директория приложения: Все файлы (БД app.db, кэш, логи) сохраняются в защищенную домашнюю папку пользователя:
* Windows: AppData/Roaming/CloakManager/
   * macOS: Library/Application Support/CloakManager/
   * Linux: .config/CloakManager/

## Структура схемы БД профиля:

* id (UUIDv4) — уникальный ключ профиля для API.
* number (INTEGER) — инкрементный порядковый номер для визуального удобства.
* name (TEXT) — пользовательское имя аккаунта.
* proxy_id (INTEGER) — внешний ключ, связывающий профиль с таблицей менеджера прокси.
* fingerprint_seed (TEXT) — уникальное зерно для генерации стабильного шума Canvas/WebGL внутри CloakBrowser.
* platform (TEXT: windows | macos | linux).
* user_agent, screen_resolution (TEXT).
* hardware_cores, hardware_memory (INTEGER).
* extensions (TEXT, JSON-массив идентификаторов используемых расширений).
* status (TEXT: stopped | starting | running).

## 4. Функциональные модули системы## 4.1. Генератор случайных отпечатков (Fingerprint Generator)

* Модуль генерации реалистичных цифровых личностей «в один клик».
* На основе выбранной platform система сопоставляет параметры, исключая логические аномалии (например, недопустимо генерировать Safari User-Agent для платформы Windows или выставлять 32 ядра процессора для конфигурации бюджетного ноутбука).
* Автоматически генерирует уникальный fingerprint_seed для маскировки Canvas/WebGL на уровне C++ ядра CloakBrowser.

## 4.2. Менеджер прокси и ротация (Proxy Manager)

* Централизованная таблица прокси (HTTP, HTTPS, SOCKS5) с возможностью пакетного импорта.
* Поддержка ротации (Мобильные прокси): В свойствах прокси закладывается поле proxy_rotation_url. Если оно заполнено, перед стартом профиля Core-движок выполняет асинхронный HTTP GET-запрос к API провайдера для смены IP, ожидает status 200 и выдерживает технологическую паузу в 3 секунды.
* Proxy Checker: После ротации (или для обычного прокси) движок делает тестовый запрос к https://ipify.org. Если прокси не отвечает или выдает ошибку, запуск браузера блокируется, статус профиля переводится в stopped, а API возвращает ошибку 412 Precondition Failed с выводом предупреждения в GUI.

## 4.3. Управление куки (Cookie Import/Export)

* API-эндпоинты для импорта/экспорта файлов куки в форматах JSON и Netscape (TXT).
* Перед запуском процесса Core-движок раскладывает куки в изолированную директорию профиля --user-data-dir.

## 4.4. Логика синхронизатора (Multi-Control)

* Модуль трансляции мыши и клавиатуры из «главного» окна в «ведомые» окна через Chrome DevTools Protocol (CDP).
* Throttling: Координаты мыши транслируются с ограничением частоты (каждые 25 мс) для минимизации нагрузки на CPU хост-системы.

## 4.5. Автоматизация ввода для ИИ (Human-like Typing)

* Специализированный метод API для ввода текста ИИ-агентом.
* Вместо мгновенной вставки значения, Core-движок через CDP эмулирует последовательное нажатие клавиш со случайной задержкой от 50 до 150 мс на символ, включая редкие случайные опечатки и их последующее удаление через Backspace.

## 4.6. Безопасная очистка дискового пространства

* Метод очистки временных файлов профиля (Cache, Code Cache, GPUCache).
* Блокировка состояния (Mutex): Очистка файлов физически заблокирована, если статус профиля в БД равен starting или running. Запрос на очистку обрабатывается только при статусе stopped. При попытке очистить кэш активного браузера API возвращает 409 Conflict.

## 4.7. Кроссплатформенный контроль процессов (Anti-Zombie)

* При старте CloakBrowser Core-движок перехватывает и сохраняет его системный PID.
* Команда STOP инициализирует штатное закрытие. Если в течение 5 секунд процесс не завершился, Core принудительно уничтожает его на уровне ОС:
* На macOS/Linux: Системный вызов process.kill(pid, 'SIGKILL').
   * На Windows: Системная команда через taskkill /F /PID <pid> (или использование Node-библиотеки tree-kill).

------------------------------
## 5. Стратегия логирования (Logging)
Система ведет раздельные логи с использованием легковесного логгера (pino):

   1. Системный лог (core.log): Фиксирует запуск API-сервера, ошибки подключения к SQLite, генерацию токенов и общие сбои бэкенда.
   2. Лог профиля (logs/profile_[ID].log): Каждый профиль имеет свой изолированный файл лога, куда записываются этапы ротации прокси, результаты проверки Proxy Checker, ошибки запуска бинарника CloakBrowser и логи сессий автоматизации ИИ.

------------------------------
## 6. Стратегия тестирования (Testing)
Для обеспечения стабильности MVP внедряется автоматическое тестирование (фреймворк Vitest):

* Unit-тесты: Валидация парсеров куки (JSON/Netscape), проверка генератора отпечатков на отсутствие логических аномалий, проверка middleware авторизации по токену.
* Integration-тесты: Тестирование изоляции SQLite (параллельная запись из разных потоков), эмуляция падения прокси с проверкой корректного срабатывания блокировки запуска, а также тест-скрипт полного цикла: запуск пустого CloakBrowser -> перехват его PID -> проверка доступности возвращаемого ws_endpoint -> принудительный kill процесса.

------------------------------
## 7. Формат ответа API для интеграции с ИИ
При успешной валидации прокси и запуске CloakBrowser, Core-движок возвращает ИИ-агенту JSON-ответ следующего формата:

{
  "status": "success",
  "profile_id": "8f3b201a-cb41-4c12-8671-50e50f3b4d11",
  "pid": 14208,
  "ws_endpoint": "ws://127.0.0.1:3000/devtools/browser/6b9a84ed-733d-4cbf-a9f4-8845fcda942b"
}

Интерпретация: ИИ-агент считывает ws_endpoint и подключает к нему свою библиотеку (Puppeteer/Playwright) для выполнения автономных задач в браузере.
