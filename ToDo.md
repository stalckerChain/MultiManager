## 4. Cross-platform Window Arranger — Замена PowerShell

### 4.1. Замена PowerShell-зависимого window arranger
Текущая реализация window arranger использует PowerShell для управления окнами, что работает только на Windows. Необходимо заменить на кроссплатформенное решение.

**Требования:**
- Поддержка Windows, macOS и Linux
- Управление позицией и размером окон браузера
- Режимы расстановки: сетка (grid) и каскад (cascade)
- Учет рабочей области (taskbar-aware WorkingArea вместо Bounds)

**Варианты реализации:**
1. Использование библиотек `node-win32-api` / `node-ffi-napi` с нативными вызовами ОС
2. Замена на `robotjs` или `nut.js` для кроссплатформенного управления окнами
3. Использование CDP `Browser.getWindowBounds` / `Browser.setWindowBounds` (если CloakBrowser их поддерживает)

## 5. Модуль миграции данных (Migration Wizard) из AdsPower и Dolphin{anty}

### 5.1. Общие сведения
Модуль предназначен для автоматизированного переноса конфигураций профилей, прокси-серверов, тегов и сессионных файлов (cookies) из сторонних антидетектов (AdsPower и Dolphin{anty}) в локальную экосистему CloakManager. Перенос осуществляется в режиме «on-premise» через локальные API-порты запущенных на ПК пользователя браузеров-доноров.

### 5.2. Функциональные требования к интерфейсу (GUI / Vue.js)

- Вкладка «Migration Wizard»: Отдельный пункт в боковом меню приложения.
- Экран выбора источника: Интерактивные карточки с логотипами AdsPower и Dolphin{anty}. При выборе отображается статус готовности локального порта (например, «AdsPower обнаружен на порту 50150» или «Браузер-донор не запущен»).
- Таблица предпросмотра (Preview Table): После успешного сканирования выводится список всех доступных для импорта профилей со следующими полями:
  - Чекбокс выбора (массовый / поштучный импорт).
  - Оригинальное имя и цветные теги профиля.
  - Привязанный прокси (если есть).
  - Наличие куки (иконка/индикатор).
- Информационная плашка (UX Warning): Обязательное отображение предупреждения:

  *«Внимание: Перенос куки и настроек выполняется успешно. Однако из-за смены ядра браузера на CloakBrowser в некоторых защищенных аккаунтах (Google, Facebook) может потребоваться повторная авторизация/ввод пароля».*

- Кнопка запуска: «Импортировать выбранные профили (X шт.)» с анимированным статус-баром (Progress Bar) в реальном времени.

### 5.3. Технические требования к бэкенду (Core API)

#### 5.3.1. Подмодуль интеграции с AdsPower

- Поиск локального API: Core-движок делает проверочный запрос к стандартному порту AdsPower.
- Сбор метаданных:
  1. Вызов эндпоинта списка профилей.
  2. Парсинг параметров: user_id (ID профиля), name (имя), domain_name (теги), user_agent, sys_app_config (разрешение экрана, ОЗУ, ядра процессора).
- Сбор прокси: Извлечение данных прокси из ответа списка профилей или через прямой запрос. Поддержка парсинга типов http, https, socks5.
- Экспорт сессий (Cookies): Для каждого выбранного user_id Core-движок выполняет запрос к API донора. Полученный JSON-массив куки валидируется и сохраняется.

#### 5.3.2. Подмодуль интеграции с Dolphin{anty}

- Поиск локального API: Проверка доступности локального порта Dolphin (по умолчанию `127.0.0.1:3001`).
- Сбор метаданных:
  1. Вызов эндпоинта списка профилей.
  2. Извлечение параметров: id, name, tags, userAgent, screen (разрешение), cpu (ядра), memory (ОЗУ).
- Экспорт сессий (Cookies):
  1. Вызов эндпоинта экспорта куки.
  2. Конвертация специфичного формата куки Dolphin в стандартный JSON-формат, совместимый с Chromium/CloakBrowser.

### 5.4. Логика сохранения и инициализации (Data Mapping)

После получения данных из стороннего API, Core-движок выполняет трансляцию параметров (Data Mapping) в собственную базу данных SQLite:

1. **Генерация Fingerprint Seed:** Поскольку сиды отпечатков AdsPower/Dolphin закрыты, CloakManager автоматически генерирует новый уникальный fingerprint_seed для каждого импортированного профиля.
2. **Запись в SQLite:** Создается новая строка в таблице profiles. Полям hardware_cores, hardware_memory и screen_resolution присваиваются значения, максимально близкие к оригинальным из браузера-донора.
3. **Развертывание Куки:** Скачанные файлы куки автоматически упаковываются и записываются по пути `AppData/Roaming/CloakManager/profiles_data/profile_[NEW_ID]/Default/Cookies`.

### 5.5. Обработка ошибок и логирование

- Все этапы запросов к сторонним API должны логироваться в `core.log` с тегом `[MIGRATION]`.
- Если в процессе скачивания куки для одного из 50 профилей произошел сетевой сбой, процесс миграции не должен прерываться. Профиль помечается в логе как `FAILED`, а движок переходит к следующему аккаунту в очереди.

### 7.2. Crypto-модуль AES-256-GCM — спецификация

> Вынесено из TASK.md при чистке. Планируется к реализации в MultiManager Ф2.

**Модуль:** `src/crypto/index.js` (новый файл)

**Зависимости:** только встроенный `crypto` модуль Node.js (npm-пакеты не нужны).

**Функции:**
- `initMasterKey()` — проверить наличие мастер-ключа в OS Keyring / system_config. Если нет — сгенерировать случайный 256-бит ключ и сохранить.
- `getMasterKey()` → Buffer (ключ в RAM).
- `encrypt(plaintext) → string` — формат `aes-256-gcm:<iv_hex>:<ciphertext_hex>:<tag_hex>`.
- `decrypt(blob) → string` — парсит формат, расшифровывает через AES-256-GCM.
- `hasEncryption() → boolean` — проверяет, есть ли мастер-ключ.
- `isEncrypted(value) → boolean` — проверяет префикс `aes-256-gcm:`.

**OS Keyring (дефолт):**
- Windows: `reg.exe ADD HKCU\Software\CloakManager /v MasterKey /t REG_SZ /d {hex_key}`
- macOS: `security add-generic-password -a CloakManager -s MasterKey -w {hex_key}`
- Linux: `secret-tool store --label=MultiManager application CloakManager masterkey {hex_key}`
- Фоллбэк при ошибке Keyring: сохранить в `system_config` (менее безопасно, но работает).

**Recovery-key:** при первой инициализации вывести в logger.info() (одноразово). Пользователь записывает вручную.

**Интеграция с queries (`src/db/queries.js`):**
- Список шифруемых колонок: `['email_password', 'twitter_password', 'twitter_auth_token', 'discord_password', 'discord_token', 'wallet_password']`
- `create(data)`: для шифруемых полей — вызвать `encrypt(value)` перед INSERT
- `getById()` / `getAll()`: для шифруемых полей — вызвать `decrypt(value)` после SELECT
- Если `hasEncryption() === false` — писать в cleartext с logger.warn() (обратная совместимость)

**Тесты:** `tests/unit/crypto.test.js` — encrypt → decrypt roundtrip, isEncrypted, формат строки.

## 6. Модуль облачной синхронизации профилей (Cloud Sync)

### 6.1. Общие сведения и логика работы
Модуль отвечает за резервное копирование и синхронизацию настроек профилей (из SQLite) и файлов сессий (папок cookies/localstorage) между компьютерами пользователя через внешние облачные хранилища.

**Что синхронизируется:** Строка настроек профиля (экспортируется в маленький JSON) + архив session.zip (содержит только файлы Cookies, Local Storage и настройки расширений из папки профиля; кэш картинок и видео игнорируется для экономии места).

**Триггеры синхронизации:**
- Upload (Загрузка в облако): Происходит автоматически через 3 секунды после закрытия профиля (статус сменился с `running` на `stopped`).
- Download (Скачивание): Происходит при нажатии кнопки «Запустить» в GUI или по API, если версия в облаке новее локальной.

### 6.2. Вариант 1: Синхронизация через Google Drive (Приоритет)

#### Функционал во Фронтенде (GUI):
- В настройках приложения появляется раздел «Синхронизация».
- Кнопка «Подключить Google Диск». При клике открывается окно браузера для стандартной OAuth2 авторизации Google.
- После успешного входа GUI отображает: «Подключено: email@gmail.com | Свободно места: X ГБ».

#### Техническая реализация на Бэкенде (Core):
1. **Авторизация:** Использование официальной библиотеки googleapis (модуль `google.auth.OAuth2`). Токены доступа (access_token и refresh_token) безопасно сохраняются в локальный SQLite.
2. **Изоляция данных:** При первом подключении движок создает в корне Google Диска пользователя скрытую/приватную папку `CloakManager_Sync_Data`.
3. **Структура папки на Google Drive:**
   - Внутри создается папка для каждого профиля по его UUID: `/CloakManager_Sync_Data/[Profile_UUID]/`
   - Внутри лежат два файла: `meta.json` (настройки) и `session.zip` (куки).
4. **Оптимизация трафика:** Перед скачиванием/загрузкой движок сверяет дату изменения файла (таймстамп). Если локальный файл новее — он перезаписывает файл на Google Диске. Если на Диске новее — скачивает его.

### 6.3. Альтернативные варианты синхронизации

#### Вариант А: S3-совместимые хранилища (Продвинутый / Командный)
Позволяет подключать любые облака (AWS S3, Backblaze B2, Cloudflare R2, Яндекс.Облако, VK Cloud).

- **Настройка:** Пользователь вводит в GUI три строчки: Access Key, Secret Key и Endpoint URL.
- **Плюс:** Идеально для командной работы, скорость загрузки выше, чем через Google Drive API.

#### Вариант Б: Telegram-бот (Бесплатный)
- **Настройка:** Пользователь создает приватного Telegram-бота, копирует Token и Chat ID в GUI.
- **Логика:** При закрытии профиля Core пакует куки в session.zip и отправляет файл через Telegram Bot API (`POST /sendDocument`). В SQLite записывается message_id. При запуске — скачивает файл из чата обратно.
- **Плюс:** Бесплатно, безлимитный объем, высокая скорость.

#### Вариант В: Локальная сетевая папка (Shared Folder)
- **Настройка:** Пользователь указывает путь к любой папке (например, `D:\MyOneDrive\BrowserSync`).
- **Логика:** При закрытии профиля программа копирует session.zip в эту папку. Клиент OneDrive/Dropbox/etc. сам синхронизирует её с облаком.
- **Плюс:** Минимум кода (простое копирование файлов), нулевой риск багов с API.

### 6.4. Контроль коллизий (Concurrency / Блокировка профиля)

Если аккаунтом пользуются два человека или два ИИ-агента на разных ПК одновременно:

- На Google Диск (или в S3) в папку профиля при запуске записывается микро-файл `lock.json`: `{"locked": true, "by": "PC-Name", "time": "..."}`.
- Если второй экземпляр попытается открыть этот же профиль, Core увидит `lock.json` в облаке, заблокирует запуск и вернет ошибку: «Профиль занят на ПК PC-Name».
- При закрытии профиля первым ПК файл `lock.json` удаляется.
