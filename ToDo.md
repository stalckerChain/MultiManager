## SOFTWARE REQUIREMENTS SPECIFICATION (SRS) / ТЕХНИЧЕСКОЕ ЗАДАНИЕ## Модуль миграции данных (Migration Wizard) из AdsPower и Dolphin{anty}## 1. Общие сведения
Модуль предназначен для автоматизированного переноса конфигураций профилей, прокси-серверов, тегов и сессионных файлов (cookies) из сторонних антидетектов (AdsPower и Dolphin{anty}) в локальную экосистему CloakManager. Перенос осуществляется в режиме «on-premise» через локальные API-порты запущенных на ПК пользователя браузеров-доноров.
------------------------------
## 2. Функциональные требования к интерфейсу (GUI / Vue.js)

* Вкладка «Migration Wizard»: Отдельный пункт в боковом меню приложения.
* Экран выбора источника: Интерактивные карточки с логотипами AdsPower и Dolphin{anty}. При выборе отображается статус готовности локального порта (например, «AdsPower обнаружен на порту 50150» или «Браузер-донор не запущен»).
* Таблица предпросмотра (Preview Table): После успешного сканирования выводится список всех доступных для импорта профилей со следующими полями:
* Чекбокс выбора (массовый / поштучный импорт).
   * Оригинальное имя и цветные теги профиля.
   * Привязанный прокси (если есть).
   * Наличие куки (иконка/индикатор).
* Информационная плашка (UX Warning): Обязательное отображение предупреждения:

«Внимание: Перенос куки и настроек выполняется успешно. Однако из-за смены ядра браузера на CloakBrowser в некоторых защищенных аккаунтах (Google, Facebook) может потребоваться повторная авторизация/ввод пароля».

* Кнопка запуска: Кнопка «Импортировать выбранные профили (X шт.)» с анимированным статус-баром (Progress Bar) в реальном времени.

------------------------------
## 3. Технические требования к бэкенду (Core API)## 3.1. Подмодуль интеграции с AdsPower

* Поиск локального API: Core-движок делает проверочный запрос к стандартному порту AdsPower: http://127.0.0.
* Сбор метаданных:
1. Вызов эндпоинта списка профилей: GET http://127.0.0.
   2. Парсинг параметров: user_id (ID профиля), name (имя), domain_name (теги), user_agent, sys_app_config (разрешение экрана, ОЗУ, ядра процессора).
* Сбор прокси: Извлечение данных прокси из ответа списка профилей или через прямой запрос GET http://127.0.0. Поддержка парсинга типов http, https, socks5.
* Экспорт сессий (Cookies):
Для каждого выбранного user_id Core-движок выполняет запрос: GET http://127.0.0. Полученный JSON-массив куки валидируется и сохраняется.

## 3.2. Подмодуль интеграции с Dolphin{anty}

* Поиск локального API: Проверка доступности локального порта Dolphin (по умолчанию http://127.0.0.1:3001).
* Сбор метаданных:
1. Вызов эндпоинта списка профилей: GET http://127.0.0.
   2. Извлечение параметров: id, name, tags, userAgent, screen (разрешение), cpu (ядра), memory (ОЗУ).
* Экспорт сессий (Cookies):
1. Вызов эндпоинта экспорта куки: GET http://127.0.0[ID_1,ID_2].
   2. Конвертация специфичного формата куки Dolphin в стандартный JSON-формат, совместимый с Chromium/CloakBrowser.

------------------------------
## 4. Логика сохранения и инициализации (Data Mapping)
После получения данных из стороннего API, Core-движок выполняет трансляцию параметров (Data Mapping) в собственную базу данных SQLite:

   1. Генерация Fingerprint Seed: Поскольку сиды отпечатков AdsPower/Dolphin закрыты, CloakManager автоматически генерирует новый уникальный fingerprint_seed для каждого импортированного профиля [3.1].
   2. Запись в SQLite: Создается новая строка в таблице profiles. Полям hardware_cores, hardware_memory и screen_resolution присваиваются значения, максимально близкие к оригинальным из браузера-донора.
   3. Развертывание Куки: Скачанные файлы куки автоматически упаковываются и записываются по пути AppData/Roaming/CloakManager/profiles_data/profile_[NEW_ID]/Default/Cookies.

------------------------------
## 5. Обработка ошибок и логирование

* Все этапы запросов к сторонним API должны логироваться в core.log с тегом [MIGRATION].
* Если в процессе скачивания куки для одного из 50 профилей произошел сетевой сбой, процесс миграции не должен прерываться. Профиль помечается в логе как FAILED, а движок переходит к следующему аккаунту в очереди.

## TO-DO / ТЕХНИЧЕСКИЕ ТРЕБОВАНИЯ: МОДУЛЬ ОБЛАЧНОЙ СИНХРОНИЗАЦИИ ПРОФИЛЕЙ (CLOUD SYNC)## 1. Общие сведения и логика работы
Модуль отвечает за резервное копирование и синхронизацию настроек профилей (из SQLite) и файлов сессий (папок cookies/localstorage) между компьютерами пользователя через внешние облачные хранилища.

* Что синхронизируется: Строка настроек профиля (экспортируется в маленький JSON) + архив session.zip (содержит только файлы Cookies, Local Storage и настройки расширений из папки профиля; кэш картинок и видео игнорируется для экономии места).
* Триггеры синхронизации:
* Upload (Загрузка в облако): Происходит автоматически через 3 секунды после закрытия профиля (статус сменился с running на stopped).
   * Download (Скачивание): Происходит при нажатии кнопки «Запустить» в GUI или по API, если версия в облаке новее локальной.

## 2. Вариант 1: Синхронизация через Google Drive (Приоритет)## Функционал во Фронтенде (GUI):

* В настройках приложения появляется раздел «Синхронизация».
* Кнопка «Подключить Google Диск». При клике открывается окно браузера для стандартной OAuth2 авторизации Google.
* После успешного входа GUI отображает: «Подключено: email@gmail.com | Свободно места: X ГБ».

## Техническая реализация на Бэкенде (Core):

   1. Авторизация: Использование официальной библиотеки googleapis (модуль google.auth.OAuth2). Токены доступа (access_token и refresh_token) безопасно сохраняются в локальный SQLite.
   2. Изоляция данных: При первом подключении движок создает в корне Google Диска пользователя скрытую/приватную папку CloakManager_Sync_Data.
   3. Структура папки на Google Drive:
   * Внутри создается папка для каждого профиля по его UUID: /CloakManager_Sync_Data/[Profile_UUID]/
      * Внутри лежат два файла: meta.json (настройки) и session.zip (куки).
   4. Оптимизация трафика: Перед скачиванием/загрузкой движок сверяет дату изменения файла (таймстамп). Если локальный файл новее — он перезаписывает файл на Гугл Диске. Если на Диске новее — скачивает его.

## 3. Альтернативные варианты синхронизации (На выбор в To-Do)
Чтобы архитектура была гибкой, в интерфейсе стоит сделать выпадающий список выбора провайдера облака. Вот 3 лучших альтернативных варианта, которые стоит добавить в To-Do:
## Вариант А: Синхронизация через S3-совместимые хранилища (Для продвинутых / Команд)
Самый профессиональный вариант. Позволяет подключать любые облака (AWS S3, Backblaze B2, Cloudflare R2, Яндекс.Облако, VK Cloud).

* Как устроено: Пользователь покупает копеечное хранилище (например, Backblaze) и вводит в GUI три строчки: Access Key, Secret Key и Endpoint URL.
* Плюс: Идеально для командной работы (арбитражники, фармеры), когда 5 человек работают с одной базой аккаунтов одновременно. Скорость загрузки через S3 в разы выше, чем через Google Drive API.

## Вариант Б: Синхронизация через Telegram-бота (Хакерский / Бесплатный способ)
Крайне популярный в узких кругах метод бесплатного хранения данных без ограничений на объем.

* Как устроено: Пользователь создает своего приватного Telegram-бота, копирует его Token и свой Chat ID в настройки GUI.
* Логика: При закрытии профиля Core-движок пакует куки в session.zip и отправляет файл документом сам себе в чат через Telegram Bot API (POST /sendDocument). В SQLite записывается message_id этого файла. При запуске профиля движок просто скачивает файл из чата обратно.
* Плюс: 100% бесплатно, безлимитный объем, не нужно регистрировать сложные Google API консоли, невероятная скорость серверов Telegram.

## Вариант В: Локальная синхронизация через Сетевую папку / Облако хоста (Shared Folder)
Самый простой вариант, не требующий вообще никакого кода интеграции API.

* Как устроено: Пользователь в настройках GUI ставит галочку «Включить локальную синхронизацию» и указывает путь к любой папке на диске Windows (например, D:\MyOneDrive\BrowserSync).
* Логика: При закрытии профиля программа просто копирует session.zip в эту папку. А дальше уже установленный в Windows клиент (OneDrive, Dropbox, Яндекс.Диск или Mega) сам своими силами синхронизирует эту папку с облаком.
* Плюс: Пишется за 10 минут (обычное копирование файлов средствами Node.js), нулевой риск багов с API.

## 4. Контроль коллизий (Concurrency / Блокировка профиля)
Критически важный пункт для To-Do, если аккаунтом пользуются два человека одновременно (или два разных ИИ-агента на разных ПК).

* Что заложить в логику: На Google Диск (или в S3) в папку профиля при запуске должен записываться микро-файл lock.json, содержащий статус {"locked": true, "by": "PC-Name", "time": "..."}.
* Результат: Если второй ИИ-агент на другом ПК попытается открыть этот же профиль, наш Core-движок увидит файл lock.json в облаке, заблокирует запуск и выдаст ошибку: «Профиль занят на ПК PC-Name». При закрытии профиля первым ПК, файл lock.json удаляется.
